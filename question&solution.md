## Phase 0:玩具图阶段(纯内存,验证控制流契约)

### #1 ✅ 关系类型在打分中被忽略,正确路径被噪声盖过

**现象**(demo 默认 query "阿里巴巴的总部在哪个省"):
- 第 1 跳正确边 `阿里巴巴-[总部位于]->杭州` 得分 0.20,被剪枝
- 噪声边 `阿里巴巴-[创始人]->马云` 得分 0.40,反被保留
- 结果:走进死胡同,找不到答案

**根因**:`score_candidates` 只把**邻居节点的描述**喂给 ranker,关系名丢了。query 里的"总部""省"应该让"总部位于"这条**边**高分,但只看 tail 节点描述时,"马云"的描述恰好与 query 有词重叠,被误判。

**解法**:打分文本改为 `f"{relation} {neighbor_desc}"`,等价于在单次打分里合并了 ToG 的 S2 关系剪枝 + S3 实体打分。

**现状**:玩具规模上提前暴露了一个会影响完整架构的设计点——这正是 demo 的价值。

## Phase 0 → Phase 1:接真组件踩到的工程坑

### #2 ✅ demo 与 graphrag_loop 的包名冲突

**现象**:`run_phase1.py` 把 demo/ 加进 sys.path 后,`from config.connections import NEO4J` 报 `'config' is not a package`。

**根因**:demo/ 下有 `config.py`(模块),graphrag_loop/ 下有 `config/`(包),两者**撞名**;Python 选了先加进 path 的那个,于是子属性找不到。

**解法**:graphrag_loop 自包含——把 demo 验证过的循环逻辑物理移植到 `loop/controller.py` `loop/beam.py` 等(契约一致),只通过 `from demo.toy_data import ...` 导入纯数据,不把 demo/ 加进 path。

**现状**:这正是"控制流契约一致"设计的真实考验——逻辑同源、物理独立,迁移代价为零。

### #3 ✅ Qdrant point id 不接受中文字符串

**现象**:`vec_store.upsert(ids=["阿里巴巴", ...])` 报错。

**根因**:Qdrant point id 只接受无符号整数或 UUID。

**解法**:用枚举整数做 id,真正的 name 放进 payload。检索时从 payload 取回。

### #4 ✅ qdrant-client 与 server 版本漂移

**现象**:`vec_store.search(...)` → `404 Not Found`。

**根因**:qdrant-client 1.18 用 `query_points` API,但 docker-compose 起的 server 1.9 不支持该 REST 端点。

**解法**:升级 Docker 镜像到 `qdrant/qdrant:v1.12.4`,client/server 对齐。**治本而非将就**——把"客户端兼容老 API"和"服务端跟上"分清,后者更稳。

### #5 ✅ τ_rel 剪枝阈值高度敏感(关键发现,已记入项目记忆)

**现象**:Phase 0 词重叠打分器分布散,τ_rel=0.25 工作良好;接 bge-reranker 后 τ_rel=0.5 几乎不剪枝,噪声只能靠 beam_width 截断。

**根因**:**bge-reranker 经 sigmoid 后,无关候选普遍聚在 0.50 附近**(sigmoid 中点效应),信号噪声距离压缩,固定阈值很难跨 query 稳定。

**解法**:`loop/beam.py` 三种剪枝模式,默认 `ratio`:
- **`ratio`(默认)**:`score ≥ 本轮最高分 × PRUNE_RATIO(0.92)` 且 `≥ floor(0.30)`。跟绝对值脱钩,跨 query 稳定。
- `gap`:相邻分数差最大处切割。**有硬伤**:分数全相等时切割点随机,可能误杀正确路径,不可做默认。
- `absolute`:老的固定阈值,保留作对比。

**现状**:Phase 2 探针实测分布(query "甄士隐家中遭遇了哪些灾难")— 信号 0.61-0.66、噪声集中 0.50,ratio 0.92 阈值线 0.607,**精准切 6/26**。**事件节点化反而提高了信号噪声区分度**(Event content 长 → 与 query 语义匹配更清晰,意外红利),原计划"收紧到 0.94-0.95"无必要,**0.92 跨 phase 稳定**。

## Phase 1 → Phase 2:抽取质量与图谱设计

### #6 ✅ LLM 抽取关系方向反、同义重复、属性当关系

**现象**(红楼梦第一回):
- `甄士隐-[女儿]->英莲` 与 `英莲-[女儿]->甄士隐` 同时出现 → 互为女儿
- `邀/邀请`、`寄居于葫芦庙/寄居于庙中` 同义未合并
- 把"姓甄名费字士隐"拆成 `姓/名/字` 三条边(属性当关系)

**根因**:这三类问题不同源——
- 方向反:LLM 能力 + prompt 没给方向规则
- 同义重复:多次抽取产生表达漂移
- 属性当关系:prompt 没明确"称谓属性不抽"

**解法**:**三管齐下,而非单点修复**——
1. **prompt 加方向规则 + 抽完自检**:"长辈/拥有者为 head"、"绝不同时抽 A→B 和 B→A 矛盾边"
2. **`extractors/alignment.py`**:LLM 分组归一(实体别名、关系同义),不破坏图结构只合并标签
3. **矛盾消解**:有 provenance 时让 LLM 按原文裁决方向,无 provenance 保守丢弃矛盾对

**关键认知**(已写在 alignment.py 注释):**对齐解决不了方向抽错本身**——那是抽取层的问题。对齐能做的是**合并同义、检测矛盾、有出处时裁决**。所以源头(prompt+模型)才是主战场,对齐是兜底。

### #7 ✅ 事件抽取召回率仅 14%,粒度时碎时空(qwen-max 关键升级)

**现象**(qwen-plus + 旧 prompt):
- 22 块文本只抽出 3 个事件(召回 14%)
- 块 12 抽 7 个碎事件("抱女欲入"+"念偈"+"消失"+"悔未问"...),粒度失控
- 块 8 完全抽空,但实际有"梦遇僧道、绛珠还泪"的核心情节

**根因**:
- 单事件限制(`event` 0 或 1 个)→ 信息密集块只抓得到主线
- 粒度判据缺失 → 模型自行决定细到动作还是粗到段落
- qwen-plus 对叙事文本的事件理解不够稳定

**解法**(双管齐下):
1. **prompt 改成多事件**:`events: [...]` 数组,允许一个 chunk 抽多个;门槛降低("只要有动作发生就算");粒度约束("以场景为单位,合并连续琐碎动作,1-3 个/块")
2. **换强模型**:.env 里 `LLM_MODEL=qwen-max`,叙事理解明显更强(块 12 自动合并了"念偈→消失→悔未问"为一个事件)

**现状**:事件数 3 → 30,有事件块占比 14% → 82%,粒度稳定 1-3。NEXT 时序链 29 条,基本还原第一回情节脉络——续写所必需。

### #8 ✅ 信息载体错位:边既要承载静态关系又要塞情节,导致图爆炸(架构级升级)

**现象/思考**:Phase 1 单层 schema 下,`甄士隐-[赠银给]->贾雨村` 这样的边,既是静态关系也藏着"中秋夜资助贾雨村赴考"的完整情节。
- 把情节塞进边 → 两个节点间出现 N 条详细边 → BFS 一跳产生海量候选 → 图爆炸
- 这正是 think-about-node.md 讨论的**边/节点设计**根本问题

**根因**:叙事文本的"关系"本质上是两类——静态拓扑(亲属/从属)和动态情节(谁做了什么),硬塞进同一种边导致信息载体错位。

**解法**:**双层 schema**(主 README §3.5)
```
节点:
  (:Entity {name, desc})              人物/地点/物 —— 薄
  (:Event  {name, content, chunk_id, order})  事件 —— 厚

边(全部薄边):
  (:Entity)-[:RELATES {type}]->(:Entity)     静态本体关系
  (:Entity)-[:PARTICIPATES]->(:Event)        参与
  (:Event)-[:NEXT]->(:Event)                 时序(续写必需)
```

**为什么契合本系统**:检索循环是"给节点打分"驱动的。事件节点化把情节落在**节点**上,正好喂给现有 cross-encoder 打分机制,无需重新设计"给边打分"。

**现状**:为终极测试(续写)铺路。NEXT 时序链是续写必需的——续写需要知道情节先后顺序。

## Phase 2 内部:循环适配、IsSup、回扩

### #9 ✅ 检索循环用不上双层图(改完 schema 才是开始)

**现象**:Phase 2 把图改成双层后,`get_neighbors` 仍能走通,但**循环代码本身还停在 Phase 1 的认知**——只看节点名分数,不读 Event content;入口召回不区分 type;证据子图平铺存放。

**根因**:schema 升级和**控制流升级**是两件事,前者只是基础。

**解法**(三处适配):
1. **入口召回双型分路**:`hybrid_recall` 改为整句召回 entity + 整句召回 event + 软链接 entity,种子带 type
2. **状态分层**:`LoopState` 加 `evidence_entities`(set)、`evidence_events`(list,带 content),controller 每轮用 `get_node_info` 批量分类填充
3. **LLM 决策**:`LLMDecisionPolicy` 把累积的 entity 集合 + event content 喂给 LLM,问"信息够不够答 query" → LLM 给出语义化的 stop/continue/置信度,不再是干瘪的"分数 ≥ θ"

**现状**:实测验证 2(灾难问题)直接以 5 个事件为种子起跳,**1 跳就收够证据**——Phase 1 单层图同样问题至少要 2-3 跳。事件节点化 + 双型召回**缩短了推理路径**。

### #10 ⚠️ 入口召回引入噪声种子

**现象**:整句向量召回会引入弱相关节点(如 "雨露""警幻仙子"),作为低权重种子混进 BFS 起点。

**根因**:整句向量是兜底召回,语义匹配宽松;软链接更精准但只覆盖明确 mention。

**解法**(部分):双型分路降低了影响——event 路用 content 向量更准。

**现状/局限**:**仍可改进**。低权重噪声进 BFS 起点后会被剪枝逐步淘汰,不致命,但浪费一跳的 LLM 调用。建议下一步:给软链接命中的种子更高优先级,整句召回的弱种子设阈值过滤。

### #11 ✅ 诚实拒答触发无效回扩(P0-1 修复)

**现象**(query "贾雨村的孩子叫什么"):
- LLM 诚实回答"原文未提供足够信息回答此问题"(没编造,这是正确行为)
- IsSup 把整段判 NONE → 触发 V_MAX=2 次回扩,每次都跑空(图里就是没这条信息)
- 浪费 2 次 LLM 调用 + 2 次 loop 循环

**根因**:IsSup 不区分"无证据捏造"(应回扩)和"诚实陈述图里没有"(应通过)。

**解法**:
1. `GeneratedAnswer.is_honest_refusal` 字段 + `_REFUSAL_PATTERNS` 短语模式("原文未提"/"无法回答"/"证据不足"等 10 个)
2. `verify_answer` 看到 `is_honest_refusal=True` 直接判 FULL 短路,不调 LLM/NLI

**现状**:实测验证(2026-06-13 log)— 该 case 回扩轮数从 2 → 0,延迟 ~80s → 37s。

### #12 ✅ 重复回扩浪费 LLM(P0-2 修复)

**现象**(同上 case 修复前):第 1 轮和第 2 轮的子查询都是 `(贾雨村的孩子的名字)`、关注实体都是 `['贾雨村']`,完全相同 → 第 2 轮必然徒劳。

**根因**:LLM 没新缺口可提时仍会被外层循环驱动重新提取。

**解法**:每轮回扩计算"指纹"`(sub_query, sorted(foci), sorted(seeds))`,与上一轮相同则中止。

**现状**:即使诚实拒答短路失效(短语集不全),指纹检测也能兜底,最多浪费一次回扩。

### #13 ⚠️ IsSup 单条证据 vs 多事实复合答案 = 假阳性(下一步治本)

**现象**(2026-06-14 log,query "甄士隐家中遭遇了哪些灾难"):
- 答案是长复合断言:"葫芦庙失火[E5];贫病交加[E1];失女患病[E3];投奔岳丈[E2][E4]"
- NLI 给整段 entail=0.94 FULL —— 但实际只是 E3(女儿失踪)蕴含了"失女患病"那一个子句,**其他三个事实根本没被独立验证**
- "取最大值"策略让"1/4 支撑"被误判为"完全支撑"

**根因**:NLI 模型对"长 hypothesis 含多个独立事实"的处理力不从心——它擅长判断"单 premise 蕴含单 hypothesis",不擅长"单 premise 部分蕴含复合 hypothesis"。当前 `verify_with_nli` 取最大值的策略,在多事实场景产生假阳性。

**主流方法**(按系统性递增):
1. **原子断言级验证(FActScore / SAFE)**—— 根本解法。把答案拆成原子事实(每条只含一个不可再分断言),逐条 NLI,聚合得分 = 支撑率。
2. **多前提联合蕴含(Microsoft GraphRAG)**—— 用 LLM 替代 NLI(LLM 天然能处理多前提)
3. **结构化生成 + 子句级验证(HippoRAG 方向)**—— 让 generator 输出 `[{claim, cited_ids}, ...]` 数组,逐条 NLI
4. **检索式核验(SAFE)**—— 把每个 atomic claim 当 query 重新检索,看能否独立找到证据(贵但最强)

**计划**:Phase 3 升级到 **方法 1 + 方法 3 组合**:让 generator 同时输出 `text`(自然语言)+ `atomic_claims`(原子断言数组),IsSup 改成对 atomic_claims 逐条验证。
- 治本:消除"取最大值"假阳性
- 副作用:需要改 generator 的输出协议,P1 测试套件相应升级

**现状/局限**:Phase 2 用 LLM 兜底处理多前提(走 `verify_with_llm` 而非 NLI 时,LLM 能给出综合判断)是简化版方法 2,够用但不是最优。当前接受这个权宜,治本留 Phase 3。

## 跨阶段关键认知(项目最重要的 5 条)

1. **控制流契约一致是迁移的安全网** —— Phase 0 在内存图上验证的循环逻辑,Phase 1 接真 Neo4j、Phase 2 引入双层 schema,**控制流代码一行未改**,只换注入的依赖。这套契约设计救了项目至少 3 次。

2. **抽取质量主战场是 prompt + 模型,对齐只是兜底** —— 同一 prompt + qwen-plus → qwen-max,事件召回率 14% → 82%。改 prompt 加约束(粒度、方向、属性禁用)效果立竿见影。alignment.py 解决不了"漏抽"和"方向抽反",只能合并同义、检测矛盾。

3. **相对剪枝优于固定阈值** —— bge-reranker sigmoid 后无关项聚 0.50,固定 τ_rel 跨 query 不稳;`ratio` 模式(`score ≥ 本轮最高分 × 0.92`)跨 phase、跨 query 稳定。事件节点化甚至意外地拉开了信号(0.61-0.66)与噪声(0.50)的距离。

4. **事件节点化的双重红利** —— 解决图爆炸是表面价值,**真正的红利**是:(a) 信息落在节点上正好喂给现有 cross-encoder 打分机制;(b) 推理路径缩短(情节问题直接命中事件起跳);(c) NEXT 时序链是续写必需的基础设施。

5. **Self-RAG 反思 token 不要凑齐四个** —— 我们刻意省了 IsUse,因为它不驱动检索流程(只对答案打有用性分),保留作"续写质量评分器"。Retrieve / IsRel / IsSup 三个就够,**关键是搭配各自的兜底**(L1 保守、IsRel 双约束终止、IsSup 诚实拒答短路 + 指纹去重 + V_MAX 兜底)。
